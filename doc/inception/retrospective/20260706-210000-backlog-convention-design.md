# Retrospective: Business-ID backlog convention with git-authoritative story files

## What happened

The owner requested a structured backlog convention with stable business IDs
(`MS<N>-E<NN>-S<NN>`), per-story git files (for grep effectiveness), and GitHub
Issues as short summaries linking to the authoritative git files. This was
designed and implemented in the final stretch of Phase 7.

## What went well

- **Git-authoritative design** — story scope lives in `doc/planning/milestones/`
  where it's grep-able, diff-able, PR-reviewable, and permanent. GitHub Issues
  are just the tracking mechanism. This makes AI-agent delivery more reliable
  (agents read files, not GitHub API).
- **Business IDs decoupled from tracker IDs** — `MS2-E3-S1` is stable regardless
  of GitHub issue number. The mapping (`gh_issue: GH-42`) lives in the story
  file's frontmatter. Branches and change artifacts use the tracker
  `workItemRef` (`GH-42`); cross-references in planning docs use the business ID.
- **Per-story files over one massive file** — the Menuvivo pattern (1717-line
  story-breakdown.md per epic) doesn't scale for grep. One file per story means
  `rg "INV-SAFE-2"` finds exactly the stories that touch that invariant.

## Improvement / pattern to propose

**Codify this as the standard ADOS backlog convention.**

1. **Template:** add `doc/templates/story-template.md` and
   `doc/templates/epic-template.md` to the ADOS framework.
2. **PM agent prompt:** update to read story files from
   `doc/planning/milestones/` and create GitHub Issues with title prefix
   `[MS<N>-E<NN>-S<NN>]`.
3. **Phase 7 activity:** "Create initial milestone backlog using the
   business-ID convention" (see `203000-backlog-missing-from-phase-7.md`).
4. **Convention doc:** every project gets `doc/planning/backlog-convention.md`
   (either installed or generated by bootstrapper).

### ID scheme summary

| Type | Format | Example |
|---|---|---|
| Story | `MS<N>-E<NN>-S<NN>` | `MS2-E3-S1` |
| Bug | `MS<N>-B<NNN>` | `MS2-B7` |
| Epic | `MS<N>-E<NN>` | `MS2-E3` |

### Folder structure

```
doc/planning/milestones/MS-<N>/
├── backlog-MS-<N>.md           # ranked overview
├── <EpicID>--<slug>/
│   ├── <EpicID>--epic.md       # epic overview
│   └── <StoryID>--<slug>.md    # one per story (authoritative scope)
└── bugs/
    └── <BugID>--<slug>.md      # one per bug
```

## Why it matters

A project without a structured backlog hands off from inception to delivery with
an implicit plan. A project with this convention hands off a grep-able,
dependency-aware, cross-cutting-tagged delivery queue that AI agents and humans
can both execute against. The business IDs survive tracker changes; the git
files survive issue archival.
